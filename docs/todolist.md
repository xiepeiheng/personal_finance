## 2. 数据展示方式设计 —— 时间范围聚合 vs 预计算字段

### 场景分析

当前分类模型的 `actual_amount` 和账本模型的 `balance` 是**全时段累计值**——从第一条交易到最后一条的总和。这在以下场景中不够用：

**场景 A：查看"2025年6月"账本的当月消费**
- 预计算：`balance` 显示的是该账本下所有分类的累计 actual_amount，是**全时段**总额
- 需求：只统计 6 月份的消费。无法用 `balance` 回答。

**场景 B：查看"2025年5月"我在"三餐"上花了多少钱**
- 预计算：`actual_amount` 是该分类全时段累计
- 需求：按月切分。无法用 `actual_amount` 回答。

### 方案

#### 方案一：时间范围实时聚合

每次查询时传入时间范围参数，后端动态聚合该范围内的交易数据。

```python
# GET /api/finance/transactions/summary/?date_from=2025-06-01&date_to=2025-06-30
# 返回:
{
  "total_income": 15000.00,
  "total_expense": 8456.90,
  "categories": [
    {"id": 1, "name": "三餐", "total": -1234.50, "count": 45},
    {"id": 2, "name": "房租", "total": -2800.00, "count": 1},
    ...
  ]
}
```

**✅ 优点：**
- 灵活，任意时间范围
- 无需维护预计算字段的一致性
- 可按自然月/自定义月（如每月 14 号）划分

**❌ 缺点：**
- 每次请求都做实时聚合，数据量大时可能慢（但个人财务数据量很小）
- 分页场景下总数不准确（如 Category 列表显示本月 actual_amount）

#### 方案二：模型关系树视图

利用数据库中已有的预计算字段，生成类似思维导图的层级结构：

```
2025年收支总账（时间账本）
├── 三餐（time）  ─────── 关联 ──→ 日常三餐（分类账本） ── 余额：-332.40
├── 交通出行（time）  ─── 关联 ──→ 出行旅游（分类账本）── 余额：-69.00
├── 房租（time）              ── 预算3000 / 实际-2800
├── 工资收入（time）          ── 实际+15000
└── 服饰（time）    ─── 关联 ──→ 服饰美容（分类账本）── 余额：-399.00
```

**✅ 优点：**
- 数据已经在数据库里，直接展示
- 一眼看清整体财务状况

**❌ 缺点：**
- 只能看全时段累计，不能按月切分
- 对于灵活的时间范围查询没有帮助

### 结论：两个方案共存，非二选一

| 使用场景 | 用哪个 |
|---------|--------|
| Dashboard 首页（本月消费柱状图、收支对比） | **方案一**：时间范围实时聚合 |
| 账本详情页（查看某账本下的分类和余额） | **方案二**：现有预计算字段 + 关系展示 |
| 月度账单 PDF/Excel 导出 | **方案一**：指定时间范围聚合 |
| 分类浏览页（快速了解各分类状态） | **方案二**：现有预计算字段 |

### 关于"自定义月划分日期"

可以做成用户自定义设置（如自然月 vs 每月 15 号作为月账单起始日），存储到用户偏好中。在时间范围查询时以此为参数。

### 关于"未关闭的人情往来分类"

当前 `is_complete` 字段恰好适用于这个场景：
- `is_complete=True` → 此分类已关闭，不再产生交易（如"2024年清明出游"）
- `is_complete=False` → 此分类活跃中，金额不为零表示仍需关注（如人情往来未还清）

可以在分类列表中用 `is_complete` 做筛选，同时在首页展示"待关闭分类"提醒（actual_amount != 0 且 is_complete=False）。
